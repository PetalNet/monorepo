# @petalnet/design (the design system page)

The living PetalNet design-system page: paper and ink, one accent, quiet by
default. It wears the system it documents.

This is a **static site** (`site/index.html` + `site/styles.css` + assets). As
of PR `janet/ds-4-daisyui-page` it is a pnpm workspace package (`@petalnet/design`)
with one build step: `site/styles.css` is **generated** from `site/app.css` (a
Tailwind v4 + daisyUI entry) by `tools/build-css.mts`. Earlier PRs in the stack
extracted the CSS (ds-1), turned the token values into a DTCG graph (ds-2), and
emitted daisyUI paper/ink themes from those tokens (ds-3).

## Layout

```
apps/design/
  site/
    index.html    the page — zero inline styles, links only styles.css
    app.css       Tailwind v4 + daisyUI ENTRY (hand-edited source)
    styles.css    GENERATED from app.css by tools/build-css.mts — do not hand-edit
    fonts/        self-hosted Geist woff2 (served at /fonts/*)
    vendor/       icons.js + prosemark.bundle.js (served at /vendor/*)
  tools/
    build-css.mts compiles app.css → styles.css (Tailwind v4 compile API)
    pixeldiff.mjs full-page screenshot diff harness (source vs this page)
```

## Tailwind v4 + daisyUI rebuild (PR janet/ds-4)

`site/app.css` rebuilds the page on **Tailwind v4 + daisyUI** instead of the
hand-written stylesheet, and compiles to `site/styles.css`:

```
pnpm --filter @petalnet/design build:css
```

How it consumes the stack:

- `@plugin "daisyui"` registers daisyUI v5; `themes: paper --default, ink
--prefersdark` enables **only** PR ds-3's two themes (the `@plugin
"daisyui/theme"` blocks, generated by `@petalnet/tokens` from the DTCG graph),
  so the role + radius variables flow from the token graph, not literals.
- A thin **alias layer** maps the page's short token names (`--petal`,
  `--surface`, `--radius`, `--s4`, …) onto daisyUI's theme variables
  (`--color-primary`, `--color-base-200`, `--radius-box`, …), plus reconstructs
  the few derived values daisyUI doesn't carry (the color-mix petal washes,
  `--elev` / `--rule-strong`, `--shadow-pop`). One bridge, so the bespoke CSS
  below stays byte-identical to the source while sourcing colors from the theme.
- The page's repeated UI is migrated into reusable **component classes** (`.btn`,
  `.card-*`, `.tile`, `.toast`, `.pm-*`, …), every metric kept pixel-exact.

### Two honest caveats (recorded, not hidden)

1. **`data-theme` name bridge.** The page addresses themes as
   `data-theme='light'|'dark'` (its anti-FOUC script + toggle persist those),
   but PR ds-3 names the daisyUI themes `paper`/`ink`. `paper` is `--default`
   (applies to `:where(:root)`), so `light` already resolves to paper; `dark`
   would NOT match `ink`, so `app.css` re-points the base color vars under
   `[data-theme='dark']` to ink's values (mirroring the token source). Cleaner
   long-term: rename the page's theme keys to `paper`/`ink`, or have ds-3 emit
   `light`/`dark` aliases.
2. **daisyUI component widgets are excluded.** The page's source-faithful class
   names (`.btn`, `.card-body`, `.toast`, `.hero`, `.skeleton`, `.toggle`, …)
   collide with daisyUI's same-named **components**, whose differing geometry
   would break pixel-identity. Since this page brings its OWN bespoke components,
   `app.css` excludes **all** daisyUI component widgets (`exclude: …;`) and uses
   daisyUI only for the theme system + base reset. The component classes are
   also emitted **unlayered** (outranking any stray daisyUI `@layer` rule).

## CSS extraction (PR janet/ds-1)

The source design page lived at `design-system/site/index.html` with a ~760-line
`<style>` block plus 27 inline `style=""` attributes scattered through the
markup. This PR moved **all** of it into `site/styles.css`:

- The entire `<style>` block was lifted verbatim into the top of `styles.css`
  and replaced in the page with `<link rel="stylesheet" href="styles.css">`.
- Each of the 27 markup `style=""` attributes was replaced with a descriptive
  utility / data class (`.mt-14`, `.chip-bg-petal`, `.space-bar-128`,
  `.curve-360-out`, `.body-ink`, etc.) defined in the new
  "EXTRACTED INLINE STYLES" block at the bottom of `styles.css`. Values are
  byte-for-byte what the inline attributes held.

After extraction the markup carries **zero** `style=""` attributes and **no**
`<style>` block.

### Known carve-out: 5 JS-generated inline styles

Five `style="..."` strings remain inside `<script>` template literals (the
swatch chip background, a tile status dot, two ProseMark line-offset
calculations, and an editor-failure fallback). These are built into the DOM at
**runtime** with **dynamic** values (an arbitrary color read from the live
theme, a `calc()` of a runtime line-height, etc.). They are not static markup
attributes and cannot be moved to a stylesheet without rewriting the JS to emit
classes; left for a later pass. They do not affect the static rendered page.

## Verifying pixel-identity

Serve both the original and this page over HTTP (the absolute `/fonts/` and
`/vendor/` paths need a docroot) and screenshot-compare:

```
# original
cd design-system/site && python3 -m http.server 8732
# extracted
cd apps/design/site && python3 -m http.server 8731
```

`tools/pixeldiff.mjs` automates this: it captures both pages full-page with
`reducedMotion: 'reduce'` (which freezes the time-dependent skeleton shimmer, as
the spec intends), at a fixed viewport, then reports the differing-pixel ratio.

```
node tools/pixeldiff.mjs http://127.0.0.1:8732/index.html http://127.0.0.1:8731/index.html /tmp/dsdiff light
```

Result for the Tailwind+daisyUI rebuild (PR ds-4): **identical page height**
(7578px both) and a residual diff of ~0.5% in **light and ink**. That residual
is a uniform **1-pixel rasterization offset in the content BELOW the live
ProseMark editor** — every DOM measurement (section tops, element boxes, font
metrics) is identical to sub-pixel precision, and the lower band realigns to a
**0-pixel** diff at `dy = −1`. It is a rendering-grid artifact of two different
stylesheets producing the same computed layout across the live editor boundary,
not a layout difference. Above the editor the diff is a true **0 pixels**.
